# PROJECT: Velxo — Production-Ready Gaming Marketplace

## ROLE & OBJECTIVE
You are the lead architect and senior full-stack engineer for Velxo (https://velxo.shop), a multi-vendor gaming marketplace startup. Your task is to design, scaffold, and implement a complete, production-ready, scalable marketplace platform. Every decision must prioritize security, performance, scalability to millions of users, and a world-class developer experience.

---

## 1. BUSINESS OVERVIEW

Velxo is a trusted marketplace connecting buyers and sellers of digital gaming products. It is NOT a simple e-commerce site — it is a secure escrow-backed marketplace with fraud detection, identity verification, dispute resolution, and AI-powered features.

### Target Users
- Buyers: Gamers purchasing accounts, coins, top-ups, gift cards, and services
- Sellers: Verified vendors listing digital gaming products
- Admins/Moderators: Platform managers handling disputes, fraud, and moderation
- Super Admins: Platform owners managing the entire ecosystem

### Competitive Advantage
- Lower fees than Carry1st and Eldorado.gg
- Faster checkout and modern UI
- Stronger security with AI-powered fraud detection
- Mobile-first design with real-time messaging
- AI-assisted search and customer support

## 4. MARKETPLACE FLOW (CRITICAL)

1. Seller registers → verifies identity (if required)
2. Seller creates listing
3. Admin reviews listing (if moderation enabled)
4. Listing goes public
5. Buyer purchases → payment held in escrow
6. Seller delivers product
7. Buyer confirms delivery
8. Funds released to seller (commission auto-deducted)
9. Buyer leaves review
10. Dispute/refund path available at any step

## 9. API DESIGN REQUIREMENTS

### RESTful Standards
- Base path: `/api/v1/`
- Resource-based URLs: `/listings`, `/orders`, `/users/me`, etc.
- Standard HTTP methods (GET, POST, PATCH, DELETE)
- Consistent response envelope: `{ success: boolean, data: any, message: string, meta?: object }`
- Pagination on all list endpoints (cursor-based preferred for scale)
- Rate limiting on all public endpoints
- API versioning strategy

### Authentication
- All protected endpoints require `Authorization: Bearer <supabase_jwt>`
- Backend validates JWT against Supabase public key
- Roles enforced via NestJS Guards
- Middleware to attach user context to requests

### Key Endpoint Groups
- `/auth/*` — session, profile (read-only from Supabase)
- `/listings/*` — CRUD, search, filter, sort
- `/orders/*` — create, track, confirm, cancel, refund
- `/escrow/*` — status, release, dispute
- `/wallet/*` — balance, transactions, withdraw
- `/messages/*` — conversations, send, read status
- `/reviews/*` — create, read, moderate
- `/admin/*` — dashboard, moderation, analytics, disputes
- `/payments/*` — webhooks, methods, transactions
- `/notifications/*` — preferences, history
- `/support/*` — tickets, responses

## 12. SECURITY REQUIREMENTS

- Input validation on ALL endpoints (class-validator + DTOs)
- SQL injection prevention (Prisma parameterized queries)
- XSS protection (Next.js built-in + CSP headers)
- CSRF protection on state-changing operations
- Rate limiting per IP and per user
- Brute force protection on auth endpoints
- File upload validation (type, size, scan for malware)
- Sensitive data encryption at rest (KYC docs, payment info)
- HTTPS everywhere
- Security headers (HSTS, X-Frame-Options, etc.)
- Dependency scanning in CI/CD
